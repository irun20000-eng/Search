8. 워커 구성과 호출 방식 · 컨텍스트 엔지니어링 · 가드레일 · 마무리 · 18:09

3줄 요약

  • 벤더 독립의 비결: 워커 호출 방식(내장/MCP/CLI/API)의 차이를 backend.json 한 곳에 모으고, 디스패처가 각 워커에 맞는 방식으로 알아서 호출 → 환경마다 최적 연결을 쓰면서 시스템은 동일하게 동작. (제미나이는 안티그래비티 CLI로 무료 사용.)
  • 컨텍스트 엔지니어링 3원칙: ① 컨텍스트 비대화 방지(context.md 1,500자·brief.md 1,200자 제한, 긴 코드는 파일 경로로만 전달) ② 필요할 때 필요한 만큼만 꺼내기 ③ 워커 결과는 다른 워커 검수 후 통과. + 브리프는 중요한 목표·제약을 앞, 금지사항을 뒤에(모델은 가운데를 잘 놓침).

TL;DR (5줄)

1. 하네스 멀티 에이전트 2.0 = 벤더 독립형 멀티 에이전트 시스템 — 클로드 코드/코덱스/안티그래비티 어디서 켜도 같은 플러그인·폴더·명령으로 동작.

2. "켜는 환경"이 지휘자(오케스트레이터)가 된다 — 클로드 코드→오퍼스, 코덱스→GPT-5.5, 안티그래비티→제미나이. 시스템은 그대로, 지휘자만 바뀐다.

3. 여러 모델이 서로 만들고 서로 검수 — 검수는 항상 다른 벤더가 맡아 자기 검수를 막고, 팬아웃/팬인으로 의견을 분배·종합한다.

4. 상태는 전부 파일로 — task.md/brief.md/context.md/result.md/learnings.md + backend.json·디스패처로 호출 방식을 추상화, 컨텍스트 엔지니어링 3원칙으로 품질 유지.

5. 안전장치 — 모든 모델 호출은 사용자 승인 후, 결과는 검수 통과 후 진행, 전 과정 로그 기록, 교훈은 누적 학습.
