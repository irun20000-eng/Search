한 줄 요약: 코드 한 줄 없이 클로드 코드(데스크톱 앱)로 "진짜 작동하는" 강사 섭외 사이트를 만든다. 핵심은 4가지 원칙 — ①재료는 파일로 ②결과를 그림 그리듯 ③하나씩 누적 ④하나씩 수정 — 과 구글 시트 연동으로 고객 문의를 자동 저장하는 것.

1. 인트로 및 결과물 공개 · 0:00

3줄 요약

  • "AI가 만들면 촌스럽다"(보라색 그라데이션·이모지 버튼)는 통념을 깨고, 돈 주고 만든 듯한 사이트를 보여준다.
  • 진짜 차이는 디자인이 아니라 작동 여부 — 허접한 사이트는 보내기 버튼이 죽어 있지만, 발표자 사이트는 문의를 넣으면 구글 시트에 한 줄 쌓인다.
  • 외주로는 50만~100만 원짜리 결과물을 직접 만든다.

핵심 인용

"그냥 장난감이냐, 진짜 사용할 수 있는 사이트냐의 차이입니다." — 0:48

내 질문

  • "작동하는 사이트"의 최소 조건은 무엇일까? → 이 영상에선 '문의 폼 + 데이터 저장(구글 시트)'으로 정의한다.

2. 바이브 코딩 시리즈 소개 · 1:04

3줄 요약

  • 누구나 쉽게 따라 하도록 시리즈로 진행: 1편(오늘) 고객 문의가 쌓이는 사이트.
  • 2편 관리자 페이지 → 3·4편 "돈 받고 사이트를 파는" 수익화까지.
  • 전제 약속: 뒤에 나오는 4가지 원칙은 하나도 건너뛰지 말 것.

핵심 인용

"오늘 1편은 고객 문의가 쌓이는 사이트를 만들 거예요. … 100만 원짜리 사이트를 여러분이 직접 만들어서 파는 수준까지 갈 거예요." — 1:04

내 질문

  • 시리즈가 "노코드 수익화"를 표방하는데, 1편의 산출물만으로 실제 판매가 가능한 수준인가?

3. 클로드 데스크톱 앱 및 Git 설치 · 1:36

3줄 요약

  • 도구는 클로드 코드 — 예전엔 까만 터미널이라 왕초보가 못 댔지만 데스크톱 앱이 그 장벽을 없앴다(말로 다 한다).
  • 준비물 ①: 클로드 데스크톱 앱(공식 사이트에서 설치·로그인). 단, 클로드 유료 요금제 필수(무료 불가).
  • 준비물 ②: Git(작업 자동 저장·되돌리기 안전벨트). 맥은 기본 포함, 윈도우는 설치 파일 다운로드.

핵심 인용

"예전엔 개발 도구가 까만 터미널 화면이라 왕초보는 손도 못 댔어요. 그런데 지금은 데스크톱 앱이 그 장벽을 없앴습니다. 말로 다 합니다." — 1:51

내 질문

  • Git을 "안전벨트"로만 쓰는데, 비개발자가 되돌리기(버전 관리)를 실제로 어떻게 트리거하나? → 영상에선 깊이 다루지 않음.

4. 원칙 1 — 재료는 파일로 던져라 · 2:37

3줄 요약

  • 흔한 실수: 자기소개·이력을 매번 프롬프트에 길게 적는 것 → 프롬프트가 한없이 길어진다.
  • 대신 내 정보를 about.md 파일 하나에 모아두고, AI에게 "이 파일 보고 알아서 가져다 써"라고 시킨다.
  • 담을 내용: 강사 소개·한 줄 포지셔닝·프로필·숫자로 보는 실적·출강 이력·강의 프로그램·연락처. 나중엔 숫자 한 줄만 바꾸고 "다시 반영해 줘"면 끝.

핵심 인용

"내 정보를 파일 하나에 싹 모아 두세요. 그리고 AI한테 이 파일 보고 알아서 가져다 써라고 시키세요." — 2:57

내 질문

  • about.md 같은 "재료 파일" 분리는 [[하네스 멀티 에이전트]]의 context.md/brief.md 분리와 같은 컨텍스트 엔지니어링 발상 아닐까?

5. 원칙 2 — 결과를 그림 그리듯 요청하기 · 3:41

3줄 요약

  • 코드를 알 필요 없이, 어떤 화면을 원하는지 디자이너에게 주문하듯 구체적으로 그려서 설명한다("위쪽을 좌우 두 칸으로 나눠줘" 등).
  • 첫 프롬프트의 숨은 꿀팁 3가지: ①역할 부여("너는 실력 있는 웹 개발자이자 디자이너야")로 결과 수준↑ ②재료는 about.md에서 ③배치를 "예쁘게"가 아니라 구체적으로.
  • 사전 팁: 파일 생성마다 묻는 허용창이 귀찮으면 권한 건너뛰기(스킵) 모드를 켜둔다. 클로드는 시키면 먼저 about.md부터 읽고 화면을 만든다.

핵심 인용

"코드 말고 결과를 그림 그리듯이 요청해라. 여러분이 코드를 알 필요가 없어요. 마치 디자이너에게 주문하듯이요." — 3:45

내 질문

  • "권한 건너뛰기"는 편하지만 위험할 수 있다 — 비개발자에게 안전한 범위는 어디까지일까?

6. 디자인 시스템 적용하기 (DESIGN.md) · 6:22

3줄 요약

  • 첫 결과물은 평범하다 — 여기서 끝내면 실패. 디자인도 파일로 던질 수 있다.
  • getdesign.md(Stripe·Vercel·Linear·Notion·Apple 등 유명 기업의 디자인 시스템을 DESIGN.md로 역공학해 모아둔 무료 사이트)에서 Stripe(결제 회사, 깔끔한 디자인으로 유명)의 규칙을 받아 design.md로 저장 + 로고 이미지도 폴더에 넣고, "이 문서를 처음부터 끝까지 그대로 적용해 줘"라고 시킨다.
  • "앞으로 모든 페이지도 이 디자인으로"라고 한 줄 박아두면 이후 페이지가 통일된 톤으로 나온다. → 감각 없어도 고수처럼 보이는 가장 빠른 길.

핵심 인용

"감각이 없는 사람이 고수처럼 보이는 가장 빠른 길입니다." — 7:23

내 질문

  • getdesign.md의 디자인은 실제 기업을 "역공학"한 것이라는데, 상용 사이트에 그대로 쓸 때 저작권·트레이드드레스 문제는 없을까?

7. 원칙 3 — 완료된 기능은 두고 하나씩 누적하며 쌓기 · 7:40

3줄 요약

  • 초보의 최대 실수는 "한 방에 다 시키기"(메인+강의+문의+디자인 동시) → AI가 헷갈려 죄다 어설퍼진다.
  • 고수는 "지금 건 그대로 두고 OO만 추가해 줘"로 하나씩 쌓는다. 메뉴만 시켜도 똑똑한 AI는 연결될 페이지까지 같은 디자인으로 자동 생성.
  • 누적 예: 출강 이력 카드 → 숫자 카운트업(0→3,400) → 카드 클릭 펼침 → 상단 흐르는 띠 → 문의 폼 동작(빈칸이면 경고) → 모바일 반응형. "디테일은 요청한 만큼 나온다."

핵심 인용

"지금 건 그대로 두고 하나씩 쌓아라. … 한 방에 다 시키면 AI가 헷갈려서 죄다 어설프게 나옵니다." — 7:45 "디테일은 우리가 요청한 만큼 나와요. 중요한 디테일은 꼭 잊지 마세요." — 10:39

내 질문

  • "하나씩 누적"은 [[하네스 멀티 에이전트]]의 "워커 결과를 검수 후 다음 단계로"와 같은 점진적 검증 철학과 통하지 않나?

8. 원칙 4 — 마음에 안 드는 부분 고치기 및 버그 수정 · 11:10

3줄 요약

  • 코드 몰라도 화면 보고 일상어로 "이거 이상해, 이렇게 해 줘"라고 하나씩 고치면 된다(간격 too far → 클로드가 직접 측정해 수정).
  • 추상적 표현도 통한다: "스트라이프 홈페이지처럼 더 고급스럽게" 한마디로 디자인이 확 좋아진다(처음에 디자인 시스템 파일을 줬기 때문).
  • AI도 실수한다(로고 사라짐·남색 박스 2개·카드 펼치면 내용 사라짐) → 당황 말고 증상을 하나씩 고쳐 달라 요청. 수정할 곳을 이미지 캡처로 보여주면 더 좋다.

핵심 인용

"스트라이프 홈페이지처럼 더 고급스럽게란 말에 디자인이 확 좋아졌어요. 추상적인 말도 통하는 거예요." — 12:20

내 질문

  • "증상을 일상어로"가 통하는 건 어디까지일까 — 기능 로직 버그도 일상어 설명만으로 고쳐질까?

9. 구글 시트 연동 — 고객 문의 자동 저장 · 13:40

3줄 요약

  • 문의를 볼 수 없으면 장난감 — 가장 쉽고 무료인 방법이 구글 시트 연동(서버 불필요).
  • 방법: 클로드에게 "문의가 구글 시트에 한 줄씩 쌓이게 해 줘. 코딩 못 하니 구글 쪽 클릭 단계를 쉬운 말로 순서대로 알려줘"라고 시킴 → AI가 절차를 안내.
  • 시트 생성 → Apps Script에 AI가 준 코드 붙여넣기 → 배포 시 액세스 권한을 '모든 사용자'로 설정(여기서 제일 많이 막힘) → 배포 URL을 클로드에 던지면 폼과 연결 완료. 테스트 문의가 시트에 한 줄 쌓인다.

핵심 인용

"배포할 때 액세스 권한을 반드시 모든 사용자로 설정해야 돼요. 여기서 제일 많이 막히거든요." — 15:28

내 질문

  • Apps Script 웹앱을 "모든 사용자" 공개로 두면 스팸·악용 위험은 없나? (문의 폼이 공개 엔드포인트가 됨)

10. 핵심 요약 및 왕초보 실행 계획 4단계 · 16:53

3줄 요약

"①재료는 파일로 던져라 ②코드 말고 결과를 그림 그리듯이 요청해라 ③지금 건 그대로 두고 하나씩 쌓아라 ④마음에 안 들면 하나씩 고쳐라. 이 네 가지가 전부예요." — 17:01

내 질문

11. 아웃트로 및 세인투 크루 6기 안내 · 18:46

3줄 요약

내 질문

  • 1편이 "혼자서도 된다"를 보여준 뒤 아웃트로에서 "혼자는 어렵다 → 크루 가입"으로 전환되는 구조 — 학습 가치와 마케팅을 어떻게 구분해서 볼까?

TL;DR (5줄)

1. 코드 0줄로 "작동하는" 사이트 — 클로드 코드 데스크톱 앱(유료 요금제 필수) + Git으로 강사 섭외 사이트를 만든다.

2. 4원칙이 핵심 — ①재료는 파일(about.md)로 ②결과를 그림 그리듯 구체적으로(역할 부여 포함) ③완료분은 두고 하나씩 누적 ④마음에 안 들면 일상어로 하나씩 수정.

3. 디자인도 파일로 — 디자인 시스템 사이트에서 Stripe 등의 규칙을 design.md로 받아 통째로 적용하면 감각 없어도 고수처럼.

4. 상태는 전부 파일로 — task.md/brief.md/context.md/result.md/learnings.md + backend.json·디스패처로 호출 방식을 추상화, 컨텍스트 엔지니어링 3원칙으로 품질 유지.

5. 안전장치 — 모든 모델 호출은 사용자 승인 후, 결과는 검수 통과 후 진행, 전 과정 로그 기록, 교훈은 누적 학습.
