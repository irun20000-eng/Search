# 클로드 확장 생태계 총정리 — 플러그인 · MCP · 스킬 · 서브에이전트 · 훅

> 「클로드 플러그인 마스터」 시리즈 **③편(개념 정리·결정 트리)**. 작성·조회 기준일 **2026-06-23 (KST)**. 다섯 확장점이 자꾸 헷갈리는 이유와 "언제 무엇을 쓰나"를 한 번에 정리한다. ①편 [[클로드 플러그인]], 선행 [[MCP(Model Context Protocol)]] 와 짝.

## TL;DR
- 클로드 코드의 확장점은 다섯이다: **슬래시 명령 · 스킬(Skill) · 서브에이전트(Subagent) · 훅(Hook) · MCP 서버**. 그리고 이 모두를 **하나로 묶어 배포하는 포장재가 플러그인**이다 [6][7][14].
- 헷갈림의 근원은 **"누가 부르나(트리거)"** 가 저마다 다르기 때문이다. 명령=사용자가, 스킬=모델이 알아서, 훅=이벤트가 자동으로, 서브에이전트=위임으로, MCP=모델이 도구로 [6][14].
- **한 줄 구분**: 명령="내가 부르는 단축 프롬프트", 스킬="모델이 필요할 때 펴보는 작업 매뉴얼", 서브에이전트="딴 방에서 일하는 부하 직원", 훅="조건 걸린 자동 스위치", MCP="바깥 세상으로 난 콘센트" [국내 비유][14].
- **결정 트리 한 문장**: 외부 연결이 필요? → MCP. 절차·지식 주입? → 스킬. 사람이 부르는 작업? → 명령. 무조건 자동 실행? → 훅. 맥락을 분리해 위임? → 서브에이전트. 위를 **묶어 나눠주고 싶다** → 플러그인 [6][14].
- **컨텍스트 비용**이 선택을 가른다. 다 켜두면 토큰을 먹는다. 스킬·서브에이전트는 "필요할 때만 로드"되어 절약형, 훅은 결정적(모델 비관여)이라 안정적이다 [12][14].

## 핵심 포인트
1. **다섯 확장점 + 한 포장재(플러그인).** 플러그인은 새로운 능력이 아니라 **나머지를 묶는 단위**다 [7][14].
2. **트리거 축으로 보면 안 헷갈린다.** 사용자/모델/이벤트/위임/도구 — 누가 발동하는지가 정체성 [6][14].
3. **결정성 축**: 훅만 결정적(항상 실행). 나머지는 모델 판단이 개입 → 신뢰성이 필요한 자리엔 훅 [14].
4. **컨텍스트 축**: 스킬·서브에이전트는 지연 로딩으로 토큰 절약, MCP는 도구 설명이 상주해 비용 발생(많이 붙이면↑) [12][14(①)].
5. **플러그인은 배포 축**: 위 넷+MCP를 합쳐 팀에 한 줄로 재현 [7].

## 1. 왜 자꾸 헷갈리나 — "기능"이 아니라 "역할"이 다르다

스킬·서브에이전트·훅·MCP·명령은 **하는 일이 겹쳐 보여서** 헷갈린다. 예컨대 "PR을 리뷰한다"는 작업은 슬래시 명령으로도, 서브에이전트로도, 스킬로도 할 수 있다. 그래서 "무엇이 맞나?"를 기능으로 물으면 답이 안 나온다.

해법은 질문을 바꾸는 것이다. **"이걸 누가, 언제, 어떤 비용으로 발동하나?"** 로 물으면 각자의 자리가 분명해진다. 국내 워크숍 후기의 비유가 출발점으로 좋다 — **"스킬은 작업 레시피, 플러그인은 (그 레시피들의) 묶음, 마켓플레이스는 저장소"** [국내 워크숍]. 여기에 트리거·결정성·컨텍스트라는 세 축을 더하면 지도가 완성된다 [14].

비유를 하나 더 들면, 클로드 코드를 **하나의 회사**라고 보자. **슬래시 명령**은 내가 직접 누르는 "단축 버튼", **스킬**은 직원들이 필요할 때 펴보는 "사내 업무 매뉴얼", **서브에이전트**는 특정 일을 전담하는 "다른 부서 직원", **훅**은 출퇴근 기록기처럼 "조건이 되면 자동으로 도는 장치", **MCP**는 외부 거래처로 연결되는 "전용 회선"이다. 그리고 **플러그인**은 이 모두를 담은 "신입 온보딩 키트"여서, 새 팀원(또는 새 머신)이 키트 하나만 설치하면 회사의 일하는 방식이 그대로 재현된다. 기능이 아니라 **이 역할 분담**으로 보면, "무엇을 쓸까"가 "누구에게 시킬까"로 자연스럽게 번역된다.

## 2. 다섯 확장점, 하나씩

### ① 슬래시 명령 (Slash Command)
- **트리거**: **사용자**가 `/명령`으로 직접 호출 [6].
- **정체**: 재사용 가능한 프롬프트/작업의 단축키. `commands/`의 마크다운 한 장.
- **쓸 때**: "내가 자주 입력하는 지시"를 박제. 예) `/commit`, `/release-notes`.

### ② 스킬 (Skill)
- **트리거**: **모델**이 상황에 맞으면 **스스로 로드**(또는 `/skill-name` 수동) [국내 스킬 3단계].
- **정체**: 특정 작업의 절차·지식·예시를 담은 파일 묶음(`SKILL.md`). description의 조건에 맞는 대화가 시작되면 클로드가 알아서 펴본다 [국내 스킬 3단계].
- **쓸 때**: "이 종류 작업은 이렇게 처리해"라는 **노하우 주입**. 평소엔 안 읽다가 필요할 때만 로드 → **토큰 절약** [14].

### ③ 서브에이전트 (Subagent)
- **트리거**: 메인 에이전트의 **위임**(또는 명시 호출).
- **정체**: **독립된 맥락·역할**을 가진 보조 에이전트. 자기 도구 세트로 한 가지 일에 집중.
- **쓸 때**: 메인 흐름을 더럽히지 않고 **분리해서 시킬 일**. 예) "버그 탐지 전용", "보안 점검 전용". 맥락이 격리돼 큰 작업에서 **컨텍스트 오염을 막는다** [12][14].

### ④ 훅 (Hook)
- **트리거**: **이벤트**(도구 사용 전/후, 세션 시작 등)에 **자동·결정적** 실행 [14].
- **정체**: 모델 판단이 아니라 **규칙으로 도는 스크립트**. "항상 일어나야 하는 일".
- **쓸 때**: 신뢰성이 필요한 자리. 예) 저장할 때마다 린트, 커밋 전 테스트, 응답 후 알림. **모델이 잊어도 무조건 실행**된다 [14].

### ⑤ MCP 서버
- **트리거**: **모델**이 노출된 **도구**로 호출(연결을 통해).
- **정체**: 외부 도구·데이터로 난 표준 콘센트(①편 주제). GitHub·Figma·Context7 등 [14(①)].
- **쓸 때**: 클로드 바깥의 세계(API·DB·파일·웹)에 닿아야 할 때. **연결이 본질**.

### (+) 플러그인 = 위 다섯을 묶는 포장재
플러그인은 새 능력이 아니라 **배포 단위**다. 명령·스킬·서브에이전트·훅·MCP를 임의 조합해 하나로 묶어 `/plugin install` 한 줄로 활성화한다 [7][14]. 더 큰 단위로 **여러 에이전트가 협업하는 "에이전트 팀(agent teams)"** 개념도 등장했는데, 이는 서브에이전트를 조직화한 상위 패턴이다 [14].

## 4. 결정 트리 — "이럴 땐 무엇을?"

```
무엇을 추가하려는가?
│
├─ 클로드 바깥(API·DB·웹·파일)에 연결해야 한다
│     └─▶ MCP 서버
│
├─ "이 작업은 이렇게 처리" 절차·지식을 모델에 알려주고 싶다
│     └─▶ 스킬 (필요할 때만 로드 → 토큰 절약)
│
├─ 내가 자주 입력하는 지시를 단축키로 만들고 싶다
│     └─▶ 슬래시 명령
│
├─ 특정 조건/시점에 "무조건" 실행돼야 한다(린트·테스트·알림)
│     └─▶ 훅 (결정적, 모델이 잊어도 실행)
│
├─ 메인 흐름과 분리해 한 역할에 집중시키고 싶다
│     └─▶ 서브에이전트 (맥락 격리)
│
└─ 위 여러 개를 묶어 팀/남에게 한 번에 나눠주고 싶다
      └─▶ 플러그인 (+ 마켓플레이스)
```

이 트리는 **상호 배타가 아니다.** 실제로는 섞인다. 예: "최신 문서로(MCP: Context7) + 우리 컨벤션대로(스킬) + 저장 시 린트(훅) + 리뷰는 전담(서브에이전트) + 호출은 `/ship`(명령)"을 **하나의 플러그인**으로 묶는 식이다. ④편 [[클로드 플러그인 워크플로우]] 의 레시피가 정확히 이 조합을 보여준다.

## 5. 컨텍스트 예산 관점 — 왜 "다 켜기"가 나쁜가

확장을 늘릴수록 좋아 보이지만, 모든 도구 설명·스킬 메타데이터는 **모델의 컨텍스트(토큰)를 먹는다**. 특히 **MCP 서버는 도구 목록·설명이 상주**해 많이 붙일수록 입력 토큰이 늘고 응답이 느려진다(①편 비용 절). 그래서 설계 원칙이 갈린다 [12][14]:

- **상시 필요 + 외부 연결** → MCP(단, 꼭 쓰는 것만).
- **가끔 필요 + 절차/지식** → 스킬(지연 로딩으로 평소 0비용에 가깝게).
- **무조건 + 가벼움** → 훅(모델을 거치지 않아 토큰·판단 비용이 거의 없음).
- **큰 작업의 한 덩어리** → 서브에이전트(별도 맥락에서 처리 후 결과만 회수).

국내·해외 가이드가 공통으로 **"3~5개만 골라라"** 라고 말하는 이유가 여기 있다 — 많이가 아니라 **맞게**가 성능이다 [12].

조금 더 구체적으로 보면, 컨텍스트는 한정된 **예산**이고 확장점마다 **과금 방식**이 다르다. MCP는 "구독료"에 가깝다 — 한 번 붙이면 도구 설명이 대화 내내 자리를 차지한다(쓰든 안 쓰든). 스킬은 "종량제"다 — 평소엔 거의 0이다가 조건이 맞을 때만 본문을 펼쳐 과금된다. 서브에이전트는 "외주"다 — 무거운 일을 별도 맥락에서 처리하고 **결과 요약만** 메인으로 가져와, 메인 예산을 아낀다. 훅은 "자동이체"처럼 모델을 거치지 않아 토큰 과금이 사실상 없다. 그래서 같은 기능이라도 **어디에 얹느냐가 비용·속도·안정성을 가른다.** 예를 들어 "사내 코딩 규칙 200줄"을 매 대화 `CLAUDE.md`에 상주시키면 늘 비싸지만, 스킬로 빼면 관련 작업일 때만 로드되어 평소 대화가 가벼워진다 [12][14]. 확장 설계란 결국 **이 예산을 어떻게 배분하느냐**의 문제다.

## 6. 흔한 오해 · FAQ

- **"스킬과 슬래시 명령은 같은 거 아냐?"** — 아니다. 명령은 **내가** 부르고, 스킬은 **모델이** 조건에 맞으면 스스로 펴본다. 자동/수동 호출의 차이가 본질 [국내 스킬 3단계].
- **"서브에이전트면 MCP 필요 없잖아?"** — 다른 축이다. 서브에이전트는 *누가 일하나*, MCP는 *무엇에 연결되나*. 서브에이전트가 MCP를 쓸 수도 있다.
- **"플러그인이 제일 좋은 거 아냐? 다 플러그인으로?"** — 플러그인은 **배포 포장**이다. 혼자 한 번 쓸 거면 명령/스킬 하나로 충분. 공유가 필요해질 때 포장한다(②편).
- **"훅이 똑똑하게 판단해주나?"** — 아니다. 훅은 **규칙대로만** 돈다. 판단이 필요하면 스킬/서브에이전트, 무조건 실행이면 훅.
- **"MCP를 많이 붙일수록 강력?"** — 반대다. 토큰·공격면이 늘어 **느려지고 위험**해진다. 꼭 쓰는 것만 [12].

## 7. 같은 작업 "PR 리뷰"를 다섯 방식으로 — 무엇이 맞나 (Worked)

확장점이 겹쳐 보이는 이유를 없애는 가장 좋은 방법은, **하나의 작업을 다섯 방식으로 짜 보고** 차이를 느끼는 것이다. "PR을 리뷰한다"를 예로 든다.

| 방식 | 어떻게 구현 | 발동 | 장점 | 약점 |
|---|---|---|---|---|
| **슬래시 명령** `/review` | 리뷰 프롬프트를 `commands/review.md`에 박제 | 내가 `/review` 입력 | 가장 단순·즉시 | 내가 매번 불러야, 매 호출이 메인 맥락 사용 |
| **스킬** | "리뷰는 이런 관점·체크리스트로"를 `SKILL.md`로 | 모델이 PR 맥락 감지 시 자동 | 알아서 적용, 평소 토큰 0 | 발동을 모델이 판단(누락 가능) |
| **서브에이전트** `reviewer` | 리뷰 전담 에이전트에 위임 | 메인이 위임 | 맥락 격리(메인 안 더럽힘), 깊게 | 왕복 비용, 설계 필요 |
| **훅** | "PR 브랜치 push 후 자동 점검" 스크립트 | 이벤트(push) 자동 | 무조건 실행(빠뜨림 0) | 유연성 낮음(규칙 고정) |
| **MCP** | GitHub MCP로 PR 변경분·코멘트 fetch | 모델이 도구 호출 | 실데이터 연결 | 연결만 제공(리뷰 *방법*은 별도) |

읽고 나면 결론은 자명하다. **이들은 경쟁이 아니라 분업이다.** 잘 만든 리뷰 플러그인은 보통 이렇게 **합친다**: MCP(GitHub로 변경분 가져오기) + 스킬(우리 팀 리뷰 기준) + 서브에이전트(보안/성능 분리 점검) + 훅(머지 전 무조건 테스트) + 명령(`/review`로 시동). 그리고 이 다섯을 **플러그인 하나로 포장**해 팀에 배포한다 [7][14]. 즉 "무엇이 맞나"가 아니라 **"어떻게 조합하나"** 가 진짜 질문이다.

## 8. 왜 확장점이 이렇게 많아졌나 — 진화의 맥락

처음엔 슬래시 명령과 `CLAUDE.md`(프로젝트 지침)뿐이었다. 그런데 쓰다 보니 한계가 드러났다. 외부 데이터에 닿질 못해 **MCP**가, 매번 같은 노하우를 붙여넣는 게 비효율이라 **스킬**이, 큰 작업에서 맥락이 오염돼 **서브에이전트**가, "모델이 잊으면 안 되는 절차"를 강제하려 **훅**이 차례로 생겼다 [9][14]. 각 확장점은 **앞선 것의 빈틈을 메우며** 등장한 셈이다.

그리고 이 조각들이 늘자 "설치·공유·재현"이 새 문제로 떠올랐고, 그걸 푸는 게 **플러그인 + 마켓플레이스**다(②편). 최근엔 서브에이전트를 여럿 조직화하는 **에이전트 팀(agent teams)** 까지 확장되며, 단일 에이전트에서 **협업하는 에이전트 집합**으로 무게중심이 옮겨가고 있다 [14]. 이 흐름을 알면 "왜 이렇게 개념이 많지?"가 "각자 다른 문제를 푸는구나"로 바뀐다.

## 9. 함정·체크리스트

**DO**
- 추가 전에 **트리거(누가 부르나)** 부터 정하기 — 그게 정체성.
- "항상 실행"은 **훅**으로(모델 신뢰 금지) [14].
- 외부 연결은 **MCP**, 단 최소한으로 [12].
- 큰 작업은 **서브에이전트**로 맥락 격리.
- 공유 시점에만 **플러그인**으로 포장(②편) [7].

**DON'T**
- 기능만 보고 고르지 말 것(겹쳐 보임) — 축으로 판단.
- MCP를 무더기로 붙이지 말 것(토큰·공격면) [12].
- 결정적이어야 할 일을 모델 판단(명령/스킬)에 맡기지 말 것 → 훅.
- 혼자 쓸 걸 플러그인부터 만들지 말 것(과설계).

## 한눈 요약
- 확장점 다섯(명령·스킬·서브에이전트·훅·MCP) + 포장재 하나(플러그인) [7][14].
- 정체성 = **트리거**(사용자/모델/이벤트/위임/도구), 선택 기준 = **결정성·컨텍스트 비용** [14].
- 결정 트리: 외부연결→MCP, 절차→스킬, 단축→명령, 무조건→훅, 격리위임→서브에이전트, 묶어배포→플러그인.
- 원칙: **많이가 아니라 맞게**, 3~5개만 [12].
