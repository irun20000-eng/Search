# 클로드 플러그인 실전 워크플로우 레시피 — 분야별 활용 사례집

> 「클로드 플러그인 마스터」 시리즈 **④편(실전 레시피·마무리)**. 작성·조회 기준일 **2026-06-23 (KST)**. ①~③편에서 "무엇을·어떻게 만드나·어떻게 구분하나"를 봤다면, 이번엔 **분야별로 끝까지 돌아가는 조합 레시피**다. 각 레시피는 *흐름도 → 쓰는 플러그인 → 단계 → 활용 포인트* 순.

## TL;DR
- 플러그인의 가치는 단일 도구가 아니라 **조합(오케스트레이션)** 에서 나온다. 이번 편은 6개 분야의 **end-to-end 레시피**를 제공한다 [13][14].
- 모든 레시피는 같은 뼈대를 공유한다: **수집 → 작성/생성 → 검수 → 저장 → 배포 → 자동화**. 분야별로 *어느 플러그인이 어느 칸에 들어가는지*만 바뀐다.
- 대표 조합: 업무(commit-commands·code-review·pr-review-toolkit), 영상→노트(YouTube/Fetch MCP·노트 플러그인), 노트(Notion·Obsidian·Drive MCP), 디자인(frontend-design·Figma), 웹(frontend-design·Playwright·Vercel·Context7) [8][9][13].
- 국내 사례가 이미 풍부하다: **Figma→Netlify 무료 배포**, **클로드 디자인으로 상세페이지**, **클로드+Vercel로 게임 만들어 배포**, **터미널 없이 웹서비스 배포** [국내 디자인/웹 사례].
- 강력할수록 위험하다. 되돌리기 어려운 칸(머지·배포·삭제·전송)엔 **승인 게이트 + 훅 검증**을 반드시 끼운다(①·③편 보안) [7].

## 핵심 포인트
1. **공통 뼈대 6칸**을 기억하면 어떤 분야든 레시피를 직접 조립할 수 있다.
2. **MCP=연결, 스킬=기준, 서브에이전트=분담, 훅=강제, 명령=시동, 플러그인=포장**(③편 요약)을 각 칸에 배치 [14].
3. **분야 경계는 흐릿하다.** 영상→노트→웹처럼 한 세션에서 분야를 가로지른다.
4. **자동화의 마지막 1%는 사람**: 핵심 분기엔 승인을 남긴다 [7].
5. 이 갤러리 자체가 살아있는 레시피다(§7) — 네이버·WebSearch·GitHub·Drive MCP의 조합 [[주제 리서치 파이프라인]].

## 0. 레시피 읽는 법 — 공통 뼈대 6칸

분야가 달라도 워크플로우의 골격은 같다. 이 6칸에 플러그인을 끼우는 게 전부다.

```
[수집] ──▶ [작성/생성] ──▶ [검수] ──▶ [저장] ──▶ [배포] ──▶ [자동화]
 MCP        모델+스킬       서브에이전트  노트 플러그인  웹/깃 플러그인  훅+명령
 (연결)     (기준 적용)     (분담 점검)   (지식 적재)    (내보내기)     (강제·시동)
```

- **수집**: 외부 데이터 끌어오기 → **MCP**(GitHub·YouTube·Fetch·Figma).
- **작성/생성**: 모델이 만든다. 품질 기준은 **스킬**로 주입.
- **검수**: **서브에이전트**(리뷰·보안·컨벤션)로 분담 점검.
- **저장**: **노트 플러그인**(Notion·Obsidian·Drive)으로 지식 적재.
- **배포**: **웹/깃 플러그인**(Vercel·Playwright·GitHub·commit-commands).
- **자동화**: **훅**(무조건 실행) + **슬래시 명령**(시동) + **플러그인**(전체 포장).

이 6칸 틀의 힘은 **"무엇을 깔지 모르겠다"는 막막함을 없앤다**는 데 있다. 플러그인 마켓플레이스를 처음 열면 수백 개가 쏟아져 압도되지만, 내 작업을 6칸으로 그려 놓으면 질문이 구체화된다 — "나는 수집 칸이 약하다(매번 자료를 손으로 찾는다) → 검색 MCP를 넣자", "검수 칸이 비었다(리뷰 없이 머지한다) → pr-review-toolkit을 넣자". 즉 **빈 칸이 곧 쇼핑 목록**이다. 반대로 이미 잘 도는 칸에 비슷한 플러그인을 또 얹는 건 과적재일 뿐이다 [12]. 그래서 레시피를 읽을 때도 "이 플러그인이 어느 칸을 채우나"를 먼저 보면 된다.

## 1. 레시피 ① 업무 자동화 — PR 리뷰·커밋 파이프라인

가장 즉효가 나는 분야다. 국내에서도 "바이브 코딩에 **딱 3개 플러그인이 게임 체인저**"로 꼽힌다 [국내 3개].

```
변경 작성 ─▶ /review(셀프 리뷰) ─▶ pr-review-toolkit(관심사별) ─▶ commit-commands ─▶ (훅) 테스트 ─▶ PR
```

- **쓰는 플러그인**: `code-review`(공식)·`pr-review-toolkit`·`commit-commands`·`security-guidance` [8][9].
- **단계**:
  1. 기능 구현 후 `code-review`로 **셀프 리뷰** → 명백한 결함 선제거.
  2. `pr-review-toolkit`의 에이전트들이 **컨벤션·버그·이력**을 분담 점검 [8].
  3. `security-guidance`가 인젝션·역직렬화·XSS 등 **취약 패턴을 인라인 지적** [8].
  4. `commit-commands`로 스테이징+컨벤셔널 메시지+커밋을 **한 턴에** [8].
  5. **훅**으로 머지 전 테스트를 무조건 실행(빠뜨림 0).
- **활용 포인트**: 리뷰·커밋의 "매번 같은 절차"를 플러그인으로 표준화하면 팀 전체 품질이 평준화된다 [국내 마켓]. 사람마다 들쭉날쭉하던 리뷰 기준이 한 플러그인으로 통일되면, 신입도 첫날부터 시니어의 점검 항목을 그대로 받는다 — 이것이 "품질의 복리"다.

## 2. 레시피 ② 영상 → 노트 (유튜브 강의 정리)

"영상 정리"는 편집이 아니라 **자막 수집 + 요약 + 저장**의 조합이다(①편 §6).

```
유튜브 URL ─▶ (MCP) 자막·챕터 수집 ─▶ 챕터별 핵심·인용 추출 ─▶ (노트) 마크다운 저장 ─▶ (선택) 블로그화
```

- **쓰는 플러그인**: YouTube/미디어 MCP·`Fetch`·노트 플러그인(Notion/Obsidian) [13].
- **단계**:
  1. MCP로 **자막·챕터·메타데이터** 수집.
  2. 모델이 **타임라인별 핵심 + 인용**을 추출(스킬로 "강의 노트 양식" 주입).
  3. 노트 플러그인으로 `YYYYMMDD_제목.md` 저장(이 갤러리 방식).
  4. (선택) `frontend-design`으로 블로그 글/페이지화 → §5로 연결.
- **활용 포인트**: 강의 10편을 **검색 가능한 지식베이스**로 바꾼다. 영상은 휘발되지만 노트는 남는다.

## 3. 레시피 ③ 노트·지식관리 — 리서치 → 옵시디언/노션

수집·작성한 것을 **단일 정본으로 쓰고 여러 곳에 분배**하는 레시피.

```
주제 ─▶ (MCP) 자료 수집 ─▶ report.md 단일 정본 ─▶ (노트) 옵시디언 + 노션 동시 적재
```

- **쓰는 플러그인**: 검색 MCP(네이버/Brave)·`Context7`(최신 문서)·Notion·Obsidian/Drive MCP [9][13].
- **단계**:
  1. 검색 MCP로 국내+해외 자료 수집(무환각: 인용 1:1).
  2. `Context7`로 최신 API/문서를 정확히 인용(환각↓) [9].
  3. `report.md` **단일 정본** 작성.
  4. 노트 플러그인으로 옵시디언 볼트 + 노션 DB에 **동시 적재**.
- **활용 포인트**: "한 번 쓰면 웹·노트·DB로 퍼지는" 구조. 정본이 하나라 **버전 충돌이 없다**. 흔한 실패는 정반대로 가는 것 — 노션에 한 번, 옵시디언에 또 한 번, 블로그에 또 한 번 손으로 옮기다 셋이 미묘하게 달라지는 "사본 지옥"이다. 단일 정본 + 플러그인 분배는 이 문제를 구조적으로 없앤다. 적재 시 **출처 메타(수집일·URL)** 를 함께 남기면 나중에 사실 검증·갱신이 쉬워지고, 위키링크로 기존 노트와 엮으면 지식이 **그래프로 누적**된다(이 갤러리의 옵시디언 운영 방식과 동일).

## 4. 레시피 ④ 디자인 → 코드 (Figma → 컴포넌트)

설치 수 1위 분야(①편). 국내에도 **"클로드 디자인으로 상세페이지 만들기"**, **"디자이너 해고각?"** 같은 실사용기가 많다 [국내 디자인].

```
Figma 시안 ─▶ (Figma MCP) 디자인 읽기 ─▶ frontend-design 컴포넌트화 ─▶ 접근성·반응형 보강 ─▶ 코드
```

- **쓰는 플러그인**: `frontend-design`(공식)·Figma MCP·디자인 시스템 도구 [9][13].
- **단계**:
  1. Figma MCP로 **시안(레이아웃·토큰)을 읽어** 구조 파악.
  2. `frontend-design`으로 **반응형 컴포넌트** 생성.
  3. 접근성(a11y)·상태(hover/disabled)·다크모드 보강.
  4. (역방향) 코드를 다시 Figma로 동기화해 디자이너와 합의.
- **활용 포인트**: 시안↔코드 **양방향**이라 "디자인 따로, 개발 따로"의 단절이 준다. 단, 브랜드 톤은 **시안 먼저 확정** 후 코드화(이 갤러리 카드 원칙과 동일).

## 5. 레시피 ⑤ 웹페이지 제작·배포 (기획 → 라이브)

"터미널 없이 웹서비스 배포"가 현실이 된 분야다 [국내 워크숍][국내 Vercel].

```
기획 ─▶ frontend-design 생성 ─▶ Context7 최신 프레임워크 ─▶ Playwright 테스트 ─▶ Vercel/Netlify 배포 ─▶ (훅) 알림
```

- **쓰는 플러그인**: `frontend-design`·`Context7`·Playwright MCP·Vercel MCP [9][13].
- **단계**:
  1. `frontend-design`으로 랜딩/상세 페이지 초안 생성 [국내 디자인].
  2. `Context7`로 Next.js 등 **최신 API**를 정확히 구현(환각↓) [9].
  3. Playwright MCP로 **핵심 플로우 E2E 테스트**(배포 전 검증).
  4. Vercel MCP로 **배포 + 런타임 로그 확인**(또는 Netlify 무료 배포) [국내 Figma+Netlify].
  5. **훅**으로 배포 후 Slack 알림.
- **활용 포인트**: 국내 사례처럼 **게임·포트폴리오·상세페이지**까지 비개발자가 띄운다 [국내 Vercel][국내 디자인]. "만들기→테스트→배포"가 한 세션에 연결되는 게 핵심. 과거엔 이 흐름의 마디마다 도구가 끊겨 있었다 — 디자인은 디자인 앱에서, 코드는 에디터에서, 테스트는 또 다른 도구에서, 배포는 터미널에서. 플러그인은 이 마디를 **한 대화 안으로 끌어와** 맥락 전환 비용을 없앤다. 그래서 "터미널 없이 웹서비스 배포"가 과장이 아니라 실제 후기로 나온다 [국내 워크숍]. 다만 배포는 되돌리기 어려운 칸이므로 **스테이징 우선 + 배포 후 Playwright 검증**을 습관화하고(§8), 도메인·환경변수 같은 운영 설정은 사람이 한 번 확인하는 게 안전하다.

## 6. 레시피 ⑥ 메타 — 이 갤러리 파이프라인 자체

이 시리즈가 올라가는 **'Search' 주제 리서치 갤러리**가 곧 살아있는 레시피다. ①~⑤를 실제로 합쳐 돌린다.

```
[수집] 네이버 MCP + WebSearch (국내·해외 균형, get_current_korean_time로 조회일 고정)
   ▼
[작성] report.md 단일 정본 (무환각·인용 1:1, DEEP 게이트 측정 wc -m/-l, grep)
   ▼
[검수] 분량·구조 객관 카운트 게이트 (하한 미달 시 머지 금지·보강)
   ▼
[배포] GitHub MCP: 커밋 → PR → main 머지 → Pages가 manifest.json 읽어 갤러리 자동 최신화
   ▼
[저장] Google Drive MCP: 옵시디언 볼트에 'YYYYMMDD_제목.md' 적재
```

- **활용 포인트**: "단일 정본 + 다중 출력"(웹+노트)을 MCP가 분배한다. 사람은 깃·파일을 손대지 않고 **클로드가 직접** PR을 만들고 노트를 저장한다. 상세 런북은 [[주제 리서치 파이프라인]].

## 7. 조합 원칙 · 안티패턴

**원칙**
- **6칸에 매핑**부터: 빈 칸을 찾아 거기 맞는 플러그인만 끼운다(필요에서 출발).
- **단일 정본**: 작성물은 하나의 원천에서 여러 곳으로 분배(버전 충돌 방지).
- **분담**: 검수는 서브에이전트로 관심사를 나눠 깊게.
- **강제는 훅, 시동은 명령**: 무조건 할 일과 내가 부를 일을 구분(③편).

**안티패턴**
- **플러그인 과적재**: 6칸이 아니라 "있어 보여서" 다 깔기 → 토큰·공격면·혼란 [12].
- **무인 맹신**: 배포·머지·전송을 승인 없이 자동화 → 사고. 핵심 분기에 사람 [7].
- **분야 칸 비우기**: 검수(서브에이전트)·강제(훅)를 빼고 생성만 자동화 → 품질·안정성 붕괴.
- **출처 불명 플러그인**: 워크플로우에 끼우는 순간 임의 코드 실행 경로가 된다(①·②편) [7].

## 8. 보안·승인 게이트 — 자동화의 안전벨트

레시피가 강력해질수록 "되돌리기 어려운 칸"이 늘어난다. 다음을 표준으로 끼운다 [7]:

| 위험 행동 | 안전장치 |
|---|---|
| PR 머지 | 사람 승인 + 머지 전 훅 테스트 |
| 배포(Vercel/Netlify) | 스테이징 먼저 + 배포 후 Playwright 검증 |
| 외부 전송(Slack/메일) | 초안 단계 + 승인 후 발송 |
| 파일 삭제/덮어쓰기 | 백업·드라이런 후 실행 |
| 비밀키 사용(MCP) | 환경변수 참조, 평문 금지 |

핵심은 **"자동화의 마지막 1%는 사람"**. 나머지 99%를 플러그인이 처리하더라도, 결과가 바깥으로 나가거나 되돌릴 수 없는 지점에는 승인을 남긴다.

## 9. 풀 워크스루 — 유튜브 강의 1편을 블로그 글로 (영상→노트→웹)

레시피 ②·③·⑤를 한 줄기로 잇는 실전 예다. 분야 경계가 흐릿하다는 말을 구체적으로 보여준다.

**상황**: 40분짜리 기술 강의 영상을, ①검색 가능한 노트로 남기고 ②블로그 글로 배포하고 싶다.

1. **수집** — YouTube MCP로 자막·챕터·설명을 가져온다. "이 영상 자막 받아서 챕터별로 끊어줘"면 충분하다. 영상이 길수록 사람이 받아쓰기엔 부담이지만 MCP는 즉시 처리한다.
2. **작성** — 모델에게 "강의 노트 양식"(스킬로 주입: 챕터 제목 → 3줄 요약 → 핵심 인용 타임스탬프 → 내 질문)을 적용시킨다. 스킬이 있으면 매번 양식을 설명할 필요가 없다(③편 컨텍스트 절약).
3. **검수** — `code-review`/사실확인 서브에이전트로 코드 스니펫·링크·용어를 점검한다. 강의 중 잘못 들린 부분이나 깨진 링크를 잡는다.
4. **저장** — Obsidian/Drive 플러그인으로 `20260623_강의제목.md`를 볼트에 적재. 위키링크로 기존 노트와 연결되어 지식그래프가 자란다.
5. **블로그화** — `frontend-design`으로 노트를 읽기 좋은 글/페이지로 재구성하고, `Context7`로 본문에 인용된 라이브러리 API를 최신으로 교정한다.
6. **배포** — Playwright로 링크·이미지가 깨지지 않는지 확인한 뒤 Vercel/Netlify로 올린다 [국내 Figma+Netlify][국내 Vercel].
7. **자동화·포장** — 여기까지가 반복된다면 `commit-commands`로 변경을 커밋하고, **훅**으로 "배포 성공 시 Slack 알림"을 걸고, 전체를 **`/video2post` 슬래시 명령을 가진 플러그인 하나로 포장**(②편)해 다음엔 명령 한 줄로 끝낸다.

> 포인트: 처음엔 7단계를 손으로 잇지만, 두 번째부터는 **플러그인 하나가 영상→노트→웹을 관통**한다. 이것이 "조합을 포장해 복리로 쓴다"는 시리즈의 결론이다.

## 10. 체크리스트

**DO**
- 워크플로우를 **6칸**으로 그려 빈 칸부터 채우기.
- 분야 가로지르기(영상→노트→웹)를 **한 세션**으로 설계.
- 검수(서브에이전트)·강제(훅)를 **반드시 포함**.
- 되돌리기 어려운 칸엔 **승인 게이트** [7].
- 잘 도는 조합은 **플러그인으로 포장**해 재사용(②편).

**DON'T**
- 6칸 매핑 없이 플러그인부터 깔지 말 것 [12].
- 생성만 자동화하고 검수·강제를 빼지 말 것.
- 승인 없이 배포·머지·전송을 무인화하지 말 것 [7].
- 출처 불명 플러그인을 워크플로우에 끼우지 말 것 [7].

## 한눈 요약
- 모든 레시피 = **수집→작성→검수→저장→배포→자동화** 6칸, 분야별로 플러그인만 교체.
- 업무·영상·노트·디자인·웹 + 이 갤러리(메타)까지 6개 레시피를 조합으로 제시.
- 가치는 단일 도구가 아니라 **오케스트레이션**, 안전장치는 **승인 게이트 + 훅** [7][13][14].
- 잘 되는 조합은 **플러그인으로 포장**해 팀에 배포(②편) → 품질의 복리.
