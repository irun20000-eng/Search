# 클로드 플러그인 완전 가이드 — 분야별 추천과 활용예시

> 「클로드 플러그인 마스터」 시리즈 **①편(개관·추천)**. 작성·조회 기준일 **2026-06-23 (KST)**. 마켓플레이스 플러그인 수·설치 수 같은 유동 수치는 모두 이 시점 기준이며 빠르게 변하므로 인용 시 날짜를 확인할 것.

## TL;DR
- **클로드 플러그인**은 **슬래시 명령(slash command) + 서브에이전트(subagent) + 훅(hook) + MCP 서버**를 하나로 묶어 **한 번에 설치하는 패키지**다. 워크플로우에 필요한 명령 3개·에이전트 2개·MCP 1개가 **단일 단위**로 배포된다 [6][7][14].
- 설치는 두 줄이다. **마켓플레이스 등록 → 플러그인 설치**: `/plugin marketplace add <owner/repo>` 후 `/plugin install <이름>@<마켓플레이스>` [1][7]. 공식 마켓플레이스 `claude-plugins-official`은 **자동 로드**된다 [7].
- 2026년 3월 기준 공식 마켓플레이스에는 **약 101개 플러그인**(Anthropic 제작 33 + 파트너 68: GitHub·Playwright·Figma·Vercel 등)이 있다 [9]. 설치 수 상위권은 **Frontend Design(82.9만)·Superpowers(75.2만)·Context7(34.8만)**(2026-06-01 집계) [9].
- **다 깔지 마라.** 공식 3종(**feature-dev·frontend-design·code-review**)이 가치의 80%를 커버한다는 평가가 많다. **내가 매주 하는 일 3~5개**에 매핑되는 것만 고른다 [9][12].
- **보안이 핵심**: 플러그인은 **내 권한으로 임의 코드를 실행**한다. 신뢰하는 출처에서만 설치하고, 조직은 **관리 설정으로 허용 마켓플레이스를 제한**할 수 있다 [7][국내 보안사례].

## 핵심 포인트
1. **플러그인 = 4-in-1 번들.** 슬래시 명령·서브에이전트·훅·MCP 서버를 임의 조합해 하나로 묶는다. 설치하면 구성요소가 **한꺼번에 활성화**된다 [6][7].
2. **마켓플레이스 = 플러그인 카탈로그(깃 저장소).** `marketplace.json`으로 목록을 관리하는 git repo다. 공식 마켓플레이스는 기본 탑재 [7][1].
3. **"분야에서 출발"한다.** 업무·자동화·영상·노트·디자인·웹 중 **내 반복 작업**부터 정하고 거기 맞는 플러그인만 [2(국내)][12].
4. **플러그인 ⊃ MCP.** MCP 서버는 플러그인이 담을 수 있는 4요소 중 하나다. 즉 플러그인은 MCP의 상위 포장 단위다(상세는 ③편) [14][6].
5. **권한은 줄수록 안전.** 플러그인은 강한 신뢰를 전제로 코드를 실행하므로, 출처·범위를 꼭 확인한다 [7].

## 1. 플러그인이란 무엇인가 — "워크플로우 한 묶음"

클로드 코드를 처음 깔면 기본기는 충분하지만, 팀·개인마다 반복하는 작업이 다르다. 누구는 PR 리뷰를, 누구는 프런트엔드 디자인을, 누구는 커밋 메시지 자동화를 매일 한다. 예전에는 이런 맞춤 기능을 **슬래시 명령 따로, 서브에이전트 따로, 훅 따로, MCP 서버 따로** 손으로 설정했다. 설정이 흩어지니 공유도, 재현도 어려웠다.

**플러그인은 이 네 가지를 한 디렉터리로 묶어 한 번에 설치하게 한다** [6][7]. 한 플러그인이 담을 수 있는 네 확장점은 다음과 같다 [7][14]:

| 구성요소 | 무엇인가 | 예 |
|---|---|---|
| **슬래시 명령** | `/명령`으로 부르는 재사용 프롬프트·작업 | `/commit`, `/review`, `/codex:setup` |
| **서브에이전트** | 특정 역할로 분기하는 보조 에이전트 | "버그 탐지 전용", "컨벤션 검사 전용" |
| **훅(hook)** | 특정 시점에 자동 실행되는 스크립트 | 커밋 전 린트, 응답 후 알림 |
| **MCP 서버** | 외부 도구·데이터 연결(①편의 주제) | GitHub·Figma·Context7 연결 |

핵심 이점: **"명령 3개 + 에이전트 2개 + MCP 1개"가 필요한 워크플로우가 단 하나의 설치 단위로 배포된다** [7]. 설치하는 순간 구성요소가 전부 켜지므로, 팀원이 똑같은 환경을 한 줄로 재현한다. 국내 안내서들도 "플러그인을 적절히 쓰면 **에이전트 활용이 무척 쉬워진다**"고 정리한다 [2(국내)].

## 2. 왜 플러그인인가 — MCP·스킬과의 자리

①편(MCP)에서 봤듯 **MCP는 "AI용 USB-C"**(외부 연결 규격)다. 그런데 MCP만으로는 "이 연결을 어떤 명령으로, 어떤 에이전트가, 어떤 시점에 쓸지"가 빠진다. **플러그인은 그 사용법까지 묶어 배포하는 상위 포장**이다 [14][6].

- **MCP 서버**: 연결(도구·데이터). 플러그인의 한 구성요소.
- **스킬(Skill)**: 특정 작업 절차·지식을 모델에 주입(파일 묶음). 플러그인으로 배포 가능.
- **플러그인**: 위 모든 것 + 명령 + 훅을 **하나로 묶은 배포 단위**.

> 이 셋(+서브에이전트·훅)의 정확한 구분과 "언제 무엇을 쓰나" 결정 트리는 본 시리즈 **③편 [[클로드 확장 생태계]]** 에서 다룬다. 여기서는 "플러그인 = 가장 큰 포장 단위"만 기억하면 된다.

## 3. 설치·관리 — 두 줄이면 끝

클로드 코드에서 플러그인은 **마켓플레이스를 등록 → 플러그인을 설치**하는 2단계다 [1][7]. 시간에 민감한 명령이므로 **공식 문서(code.claude.com/docs) 기준**으로 확인하고 쓸 것 [6].

```bash
# 1) 마켓플레이스(플러그인 카탈로그 git repo) 등록
/plugin marketplace add <owner/repo>

# 2) 플러그인 설치 (이름@마켓플레이스)
/plugin install <플러그인이름>@<마켓플레이스>

# 예: 커밋 자동화 플러그인
/plugin install commit-commands@claude-code-plugins
```

- **공식 마켓플레이스 `claude-plugins-official`은 자동 로드**되므로 별도 등록 없이 바로 설치할 수 있다 [7].
- 관리: `/plugin` 메뉴에서 설치 목록·활성/비활성·제거를 다룬다.
- **마켓플레이스의 정체**: 단순한 git 저장소이며, `marketplace.json`이 플러그인 목록(이름·설명·경로)을 카탈로그한다 [7]. 그래서 **누구나 자기 마켓플레이스를 만들어 배포**할 수 있다(→ ②편 [[클로드 플러그인 만들기]]).

국내 입문 후기들도 이 흐름을 그대로 보여준다. 한 글은 "Superpowers와 OMC는 **명령어 한 줄로 설치가 끝나고**, BKIT은 마켓플레이스로 받는다"고 정리했고 [3(국내)], 초보자용 따라하기 글도 마켓플레이스 등록 → 설치 순서로 안내한다 [1(국내)].

```
설치 흐름 한눈에
┌────────────────────────┐     ┌──────────────────────────┐
│ /plugin marketplace add │ ──▶ │ marketplace.json 카탈로그 │
│   owner/repo            │     │  (플러그인 목록 등록)     │
└────────────────────────┘     └─────────────┬────────────┘
                                              │ /plugin install name@market
                                              ▼
                        ┌──────────────────────────────────┐
                        │ 슬래시명령 + 서브에이전트 +       │
                        │ 훅 + MCP 서버  ── 한꺼번에 활성화 │
                        └──────────────────────────────────┘
```

## 4. 분야별 추천 ① 업무·생산성

코드 리뷰·기획·문서 같은 **일상 업무 묶음**이다. 공식·검증 플러그인부터 시작하는 게 안전하다 [9][12].

| 플러그인 | 무엇을 하나 | 활용예시 |
|---|---|---|
| **feature-dev**(공식) | 기능 개발 워크플로우(설계→구현→점검) 묶음 | "이 이슈를 기능 단위로 쪼개 구현·자체 점검까지" |
| **code-review**(공식) | 변경분 리뷰 에이전트 | PR 올리기 전 셀프 리뷰로 결함 선제거 |
| **pr-review-toolkit** | 관심사별 리뷰 에이전트(컨벤션·버그·이력) | CLAUDE.md 규약 준수·버그·히스토리 점검을 분담 |
| **commit-commands** | 스테이징+컨벤셔널 커밋 메시지+커밋을 한 턴에 | `/commit` 한 번으로 정리된 커밋 생성 |

국내 매체는 마켓플레이스를 "**클로드를 엔진으로 쓰는 앱·플러그인을 모아 둔 곳**"으로 소개하며, 업무 효용으로 **코드 리뷰·테스트 케이스 생성·기획서 초안·경쟁사 분석**을 든다 [4(국내)]. 즉 개발 외 기획·리서치 업무도 플러그인으로 표준화된다.

## 5. 분야별 추천 ② 자동화

반복을 사람이 아니라 **에이전트가 돌리게** 하는 묶음이다.

| 플러그인 | 무엇을 하나 | 활용예시 |
|---|---|---|
| **Ralph Loop** | 자율 코딩 세션(작업을 하나씩 구현·커밋) | CRUD·마이그레이션·테스트 커버리지 같은 반복 작업 자동 진행 [12] |
| **Superpowers** | 강력한 명령·자동화 묶음(설치 수 상위) | 자주 쓰는 멀티스텝 작업을 명령 한 줄로 [9][3(국내)] |
| **commit-commands** | 커밋 파이프라인 자동화 | 변경 정리→메시지→커밋 무인화 |
| **(MCP 기반) GitHub 플러그인** | PR·이슈·CI를 대화로 자동 처리 | "테스트 통과하면 머지" 같은 흐름 |

> 자동화는 강력한 만큼 위험하다. 되돌리기 어려운 행동(머지·삭제·외부 전송)은 **승인 게이트**를 두고, 훅으로 **사후 검증**을 거는 게 좋다(상세 ④편 [[클로드 플러그인 워크플로우]]).

## 6. 분야별 추천 ③ 영상 정리·미디어

영상 자체 편집은 클로드의 본령이 아니지만, **자막·대본·메타데이터 정리**는 MCP 기반 플러그인으로 강력하다(이 갤러리도 유튜브 MCP로 자막을 끌어온다).

| 도구(플러그인/MCP) | 무엇을 하나 | 활용예시 |
|---|---|---|
| **YouTube/미디어 MCP 플러그인** | 영상 자막·대본·댓글·통계 수집 | "이 강의 영상 자막을 받아 챕터별 요약 노트로" |
| **Fetch/웹 수집 계열** | 영상 설명·관련 페이지 수집 | 영상→블로그 초안 변환 |
| **노트 연동 플러그인(아래 ④)** | 정리 결과를 옵시디언/노션에 저장 | 영상 요약을 지식베이스로 적재 |

활용예시(이 갤러리 실사례): **유튜브 자막 수집 → 핵심 타임라인·인용 추출 → 마크다운 노트로 저장**. "영상 정리"는 사실상 **수집(MCP) + 요약(모델) + 저장(노트 플러그인)** 의 조합이다.

## 7. 분야별 추천 ④ 노트·지식관리

"수집·작성한 것을 어디에 남기나"를 푸는 묶음이다. 이 갤러리의 옵시디언 저장이 대표 사례다.

| 도구(플러그인/MCP) | 무엇을 하나 | 활용예시 |
|---|---|---|
| **Notion MCP 플러그인** | 페이지·DB 읽기/쓰기 | 회의록·리서치를 노션 DB에 자동 적재 |
| **Obsidian 연동(REST/MCP)** | 볼트에 마크다운 노트 생성 | 보고서를 `YYYYMMDD_제목.md`로 저장(이 파이프라인) |
| **Google Drive MCP 플러그인** | 드라이브에 파일 생성·검색 | 클라우드 세션에서 옵시디언 볼트로 저장 |
| **Context7** | 최신 라이브러리 문서 주입 | "노트 쓸 때 최신 API를 정확히 인용" [9] |

활용예시: **리서치 → report.md 단일 정본 → (갤러리 + 옵시디언) 동시 배포**. 노트 플러그인이 "한 번 쓰면 여러 곳에 퍼지는" 분배를 맡는다.

## 8. 분야별 추천 ⑤ 디자인

디자인은 클로드 플러그인 생태계에서 **설치 수 1위 분야**다(Frontend Design 82.9만) [9].

| 플러그인 | 무엇을 하나 | 활용예시 |
|---|---|---|
| **frontend-design**(공식) | UI/프런트엔드 디자인 워크플로우 | "이 와이어프레임을 반응형 컴포넌트로" |
| **Figma MCP 플러그인** | 디자인↔코드 양방향(디자인 읽기/생성) | Figma 시안을 코드로, 코드를 Figma로 |
| **(디자인 시스템 도구)** | 토큰·컴포넌트 라이브러리 연동 | 브랜드 토큰을 일관 적용 |

활용예시: **Figma 시안 읽기 → 컴포넌트 코드 생성 → frontend-design으로 반응형·접근성 보강**. 이 갤러리의 라이트/웜화이트 카드 디자인도 같은 결로 시안을 먼저 확정한 뒤 코드화한다.

## 9. 분야별 추천 ⑥ 웹페이지 제작

"만들고 → 테스트하고 → 배포"까지 한 줄기로 잇는 묶음이다.

| 플러그인 | 무엇을 하나 | 활용예시 |
|---|---|---|
| **frontend-design**(공식) | 페이지·레이아웃 생성 | 랜딩페이지 초안 생성 |
| **Playwright MCP 플러그인** | 실제 브라우저로 조작·E2E 테스트 | "배포 전 핵심 플로우 자동 테스트" |
| **Vercel MCP 플러그인** | 배포·런타임 로그 확인 | "이 프로젝트를 Vercel로 배포하고 로그 확인" |
| **Context7** | 프레임워크 최신 문서 주입 | Next.js 최신 API로 정확히 구현 [9] |

활용예시(end-to-end): **frontend-design로 페이지 생성 → Playwright로 테스트 → Vercel로 배포 → 로그 확인**. 이 네 플러그인이 "기획자도 웹을 띄우는" 흐름을 만든다(④편에서 레시피로 상세화).

## 10. 활용 예시 (Worked) — 분야를 가로지르는 시나리오

**시나리오: "유튜브 강의 한 편을 블로그 글로 만들고 배포"**

```
[수집]   YouTube MCP 플러그인 → 자막·챕터 수집
[정리]   모델이 챕터별 핵심 + 인용 타임라인 추출
[작성]   frontend-design → 글 레이아웃/페이지 초안
[검수]   code-review/security-guidance → 링크·코드 스니펫 점검
[저장]   Notion/Obsidian 플러그인 → 지식베이스에 노트 적재
[배포]   Vercel 플러그인 → 페이지 배포, Playwright로 링크 검증
[자동]   commit-commands → 변경 커밋, 훅으로 배포 후 알림
```

여기서 핵심: **단일 플러그인이 아니라 "여러 플러그인의 오케스트레이션"이 가치를 만든다.** 영상(수집)·디자인(레이아웃)·노트(저장)·웹(배포)이 한 세션에서 이어진다. 이게 플러그인 생태계가 노리는 그림이다 [12][14].

## 11. 보안·주의 — 가장 중요한 절

플러그인은 편의가 큰 만큼 **위험도 크다**. 핵심 사실: **플러그인은 내 권한으로 임의 코드를 실행한다** [7].

- **신뢰 경계**: "신뢰하는 출처에서만 설치하라"는 게 공식 권고다. 마켓플레이스가 git repo이므로 **누구나 올릴 수 있다** [7].
- **조직 통제**: 기업은 **관리 설정(managed settings)으로 허용 마켓플레이스를 제한**할 수 있다 [7].
- **실제 위협 사례**: 2026년 6월, 한 IDE 마켓플레이스에서 **악성 플러그인이 무더기로 유포**된 사건이 보고됐다(개발 환경에 설치 여부 즉시 점검 권고) [국내 보안사례]. 마켓플레이스형 생태계 공통의 리스크다.
- **방어 원칙**: ①공식/검증 배포처 우선 ②설치 전 구성요소(특히 훅·MCP) 확인 ③되돌리기 어려운 행동엔 승인 게이트 ④비밀키는 환경변수 ⑤정기 점검·제거.

## 12. 베스트 프랙티스 체크리스트

**DO**
- 분야에서 출발: 내가 매주 하는 일 3~5개부터.
- 공식 3종(feature-dev·frontend-design·code-review)으로 시작 [9].
- 마켓플레이스·플러그인 출처를 확인하고 신뢰처만.
- 되돌리기 어려운 자동화엔 승인 게이트·훅 검증.
- 유동 수치(설치 수·플러그인 수)는 조회일과 함께.

**DON'T**
- 마켓플레이스의 모든 플러그인을 깔지 말 것(토큰·공격면·혼란↑) [12].
- 출처 불명 마켓플레이스 등록 금지(임의 코드 실행) [7][국내 보안사례].
- 훅·MCP 구성요소를 안 보고 설치하지 말 것.
- "자동화=무인" 맹신 금지 — 핵심 분기엔 사람이 개입.

## 한눈 요약
- 플러그인 = **슬래시명령+서브에이전트+훅+MCP**를 묶은 **한 번에 설치하는 워크플로우 단위** [6][7].
- 설치: `/plugin marketplace add` → `/plugin install name@market`, 공식 마켓플레이스 자동 로드 [1][7].
- 분야별: 업무(feature-dev·code-review·pr-review-toolkit) / 자동화(Ralph Loop·Superpowers·commit-commands) / 영상(YouTube·Fetch MCP) / 노트(Notion·Obsidian·Drive) / 디자인(frontend-design·Figma) / 웹(frontend-design·Playwright·Vercel·Context7).
- **다 깔지 말고 3~5개**, **신뢰처에서만**(임의 코드 실행) [7][12].
