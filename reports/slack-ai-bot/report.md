# AI 봇 만들기 — LLM+Slack으로 무엇을 할 수 있고, 1인에게 유익한가

> 작성·조회 기준일 **2026-06-24 (KST)**. 모델명·요금·기능은 시간에 민감하므로 인용 시 날짜 확인. Slack 3부작의 ③편(앞: [[Slack]] 개요편 · 봇 실전편). 이번엔 봇에 **LLM(클로드)**을 붙여 "무엇을 할 수 있는지"와 "나에게 이득인지"를 정면으로 다룬다.

## TL;DR
- **규칙 봇**(앞 편)은 "정해진 대로"만 한다. **AI 봇**은 자연어를 이해하고 **판단·요약·생성**까지 한다 — Slack 채널 안에서 일하는 **IC급(실무자급) 보조 직원**에 가깝다 [6][9].
- 1인에게 즉시 유익한 5가지: **①요약(긴 스레드·문서·회의)** ②**지식 Q&A**(내 문서·코드 기반) ③**분류·트리아지**(문의/에러 분류·우선순위·티켓화) ④**운영 에이전트**(에러 요약·원인 후보·액션) ⑤**개인 업무 도우미**(초안·번역·정리) [6][7][8].
- 만드는 법은 의외로 단순: **Bolt(앞 편) + 클로드 API**. 멘션을 받으면 텍스트를 클로드에 넘기고 답을 채널에 올린다. 핵심 함정은 **Slack 3초 타임아웃** — 먼저 `ack`/대기 메시지를 보내고 **비동기로** 답한다 [12][13].
- "나에게 유익한가"의 판정 기준: **(a) 반복 빈도 (b) 정보 과부하 (c) 즉답 가치 (d) 오류 허용도**. 네 개 중 둘 이상이 높으면 만들 가치가 있다. 반대로 가끔·정확성 100% 필수·맥락이 매번 다른 일은 봇이 손해다.
- 공짜가 아니다: **토큰 비용, 환각, 권한·프롬프트 인젝션**을 설계에 반영해야 한다 [9][10]. 그래도 **첫 봇으로 "요약 봇"**은 거의 항상 본전을 뽑는다.

## 핵심 포인트
1. **규칙 봇 → AI 봇**: "if문"이 아니라 **언어 이해+판단**이 더해진다 [6].
2. **Slack = AI의 입출력 창구**, 두뇌는 클로드 API, 실제 일은 내 코드/도구 [11].
3. **3초 타임아웃**이 최대 함정 — 즉시 ack 후 비동기 응답(스트리밍은 `chat.update`) [13].
4. **유익함은 작업 성격이 결정** — 반복·과부하·즉답·오류허용 4축으로 판정.
5. **첫 봇은 작게**: "이 스레드 요약해줘"부터. ROI가 가장 확실 [7].

## 0. 한 줄 답 — "봇으로 뭘 할 수 있는데?"

직답부터. **AI 봇은 Slack 채널 안에 상주하는 "24시간 일하는 실무자급 보조"**다. 2026년 가이드들의 표현을 빌리면, "**IC(개인 기여자)급 직원처럼 Slack 채널 안에서 동작하는 AI 에이전트**"를 만들어 "빠른 답과 똑똑한 행동이 필요한 거의 모든 반복 작업"을 맡길 수 있다 [9]. 즉 봇은 *장난감*이 아니라 **사람을 못 늘리는 1인이 레버리지를 얻는 수단**이다. 다만 "모든 일"에 유익한 건 아니다 — 그 경계를 §10에서 정직하게 가른다.

## 1. 무엇이 달라지나 — 규칙 봇 vs AI 봇

앞 편(봇 실전)에서 만든 건 **규칙 봇**이다. "ping이 오면 pong"처럼 정해진 입력에 정해진 출력. AI 봇은 여기에 **언어 이해와 판단**을 더한다 [6].

| 구분 | 규칙 봇(앞 편) | AI 봇(이번 편) |
|---|---|---|
| 입력 처리 | 정확히 일치(키워드·명령) | **자연어**(모호해도 의도 파악) |
| 출력 | 고정 템플릿 | **상황에 맞게 생성**(요약·답변·초안) |
| 판단 | 없음(if문) | **분류·우선순위·추론** |
| 예 | `/status` → 고정 응답 | "@봇 어제 #errors 요약해줘" → 핵심만 정리 |
| 두뇌 | 내 코드 | 내 코드 + **LLM(클로드)** |

핵심: AI 봇이라도 **실제 행동(배포·DB 조회)은 여전히 내 코드**가 한다. LLM은 "이해·판단·문장 생성"을 담당하는 **두뇌**일 뿐이다. 그래서 구조는 *Slack(입출력) + 클로드(두뇌) + 내 코드/도구(손발)*다 [11].

## 2. 할 수 있는 일 ① 요약 — 정보 과부하를 없앤다

가장 ROI가 확실한 용도다. 긴 스레드·문서·회의록을 **핵심만** 추려준다. Slack 자신도 "요약은 정보 과부하를 정복하는 길"이라며 밀고 있다 [10][7].

- **스레드 요약**: "이 100개짜리 스레드 결론만" → 3줄 요약 + 결정사항.
- **놓친 채널 따라잡기**: 며칠 자리를 비운 사이 쌓인 글을 한 번에 [7].
- **회의·문서 요약**: 회의록·긴 문서를 요점·할 일로. 국내에서도 "AI 회의록·메모 정리"가 인기 활용이다 [국내 회의록].

1인에게 의미: 혼자라 **모든 정보를 직접 읽어야 하는** 사람일수록 요약 봇의 시간 절약이 크다.

## 3. 할 수 있는 일 ② 지식 Q&A — 내 자료로 답하는 봇

"우리 문서/코드/위키에 근거해 답하는" 봇이다. 승인된 출처를 검색해 **정확한 답**을 만든다(이른바 지식봇/AI 검색 어시스턴트) [6].

- **예**: "@봇 환불 정책이 어떻게 되지?" → 내 정책 문서를 근거로 답 + 출처 링크.
- **개발용**: "이 함수 어디서 쓰여?" → 코드베이스 기반 답변.
- **주의**: 근거 없이 지어내면(환각) 독이다. **출처를 함께 제시**하고, 모르면 "모른다"고 하도록 설계해야 한다(§11).

이게 발전하면 RAG(검색증강생성)·MCP로 **실시간 문서 연결**까지 간다(→ [[MCP(Model Context Protocol)]]).

## 4. 할 수 있는 일 ③ 분류·트리아지 — 들어온 걸 정리

문의·에러·요청이 들어오면 **분류하고 우선순위를 매겨** 다음 단계로 넘긴다. 2026 가이드의 대표 시나리오다 [6][9].

```
들어온 Slack 메시지(문의/버그) 
   │  ① LLM이 내용 이해
   ▼
[분류] 버그? 문의? 기능요청?   [우선순위] 긴급/보통/낮음
   │
   ▼
[행동] 티켓 생성 → 담당(나)에게 알림 → 상태 추적
```

"커스텀 에이전트가 Slack 지원 요청을 해석해 **긴급도를 분류하고, 티켓을 만들고, 엔지니어에게 알리고, 해결을 추적**한다"는 게 표준 패턴이다 [6]. 1인에게는 **고객 문의 1차 분류**나 **에러 자동 분류**가 특히 유용하다.

## 5. 할 수 있는 일 ④ 운영 에이전트 — 보고 + 행동

앞 편의 "운영 상황실"을 AI로 끌어올린다. 단순 알림을 넘어 **이해하고 행동을 제안/실행**한다.

- **에러 요약·원인 후보**: `#errors`에 예외가 뜨면 봇이 **로그를 요약하고 원인 후보**를 스레드에 단다.
- **데이터 질문**: "@봇 어제 매출 얼마야?" → DB 조회(내 코드) + 자연어 답변. 클로드 쿡북의 "Slack 데이터 분석 봇"이 이 패턴이다 [11].
- **행동 실행**: "배포해줘" → (확인 버튼) → 배포 스크립트 실행. **되돌리기 어려운 행동엔 승인 게이트**(앞 편 철학).

여기서 LLM은 "무엇을 할지" 판단하고, **실제 실행은 내 함수/도구**가 한다. 도구를 쓰는 단계가 §9의 에이전트다.

## 6. 할 수 있는 일 ⑤ 개인 업무 도우미

채널 협업이 아니어도, **나 혼자 쓰는 비서**로 훌륭하다. 국내 AI 챗봇 입문서들이 1장부터 "**나만의 개인 업무 도우미 챗봇**"을 꼽는 이유다 [국내 챗봇책][국내 도우미].

- **초안 작성**: 이메일·릴리스 노트·공지 초안을 Slack에서 바로.
- **번역·교정**: 외국어 문의 번역, 문장 다듬기 [국내 업무활용].
- **스탠드업/체크인**: 봇이 DM으로 "오늘 뭐 할 거야?"를 묻고 답을 모아 정리 [6].
- **메모·할 일 정리**: 던져둔 메모를 봇이 분류·요약 [국내 회의록].

## 7. 어떻게 붙이나 — 아키텍처와 3초 함정

구조는 단순하다: **Slack(Bolt) ↔ 내 서버 ↔ 클로드 API**.

```
사용자 @봇 멘션
   │  (Slack → Bolt, Socket Mode)
   ▼
내 코드: 텍스트 추출 → (필요시 DB/문서 조회) → 클로드 API 호출
   │                                   ▲ 두뇌(이해·생성)
   ▼
클로드 응답 → say()/chat.update 로 채널에 게시
```

⚠️ **최대 함정 = Slack 3초 타임아웃**. 슬래시 명령·액션은 3초 안에 `ack`해야 하는데, **클로드의 첫 토큰까지 보통 0.8~2초**, 가끔 더 걸려 3초를 넘긴다 [13]. 해법:
1. **즉시 `ack()`** 하고 "⏳ 생각 중…" 임시 메시지를 올린다.
2. 백그라운드로 클로드 호출(**lazy-listener 패턴**) [13].
3. 답이 오면 `chat.update`로 임시 메시지를 **교체**(스트리밍이면 토막마다 update) [13].

Socket Mode(앞 편)면 이벤트(`app_mention`)는 비교적 여유가 있어 1인 개발 환경에 잘 맞는다.

## 8. 최소 코드 — Bolt + 클로드 API

앞 편의 Bolt 봇에 클로드를 끼우면 끝이다. (Python, 2026-06 기준 모델 `claude-opus-4-8`) [12][14]

```bash
pip install slack_bolt anthropic
```

```python
import os
from slack_bolt import App
from slack_bolt.adapter.socket_mode import SocketModeHandler
from anthropic import Anthropic

app = App(token=os.environ["SLACK_BOT_TOKEN"])   # xoxb
claude = Anthropic()                              # ANTHROPIC_API_KEY 환경변수

@app.event("app_mention")
def on_mention(event, say):
    # 1) 즉시 대기 메시지(3초 함정 회피)
    placeholder = say(text="⏳ 생각 중…", thread_ts=event["ts"])

    # 2) 클로드에 질문 전달 (멘션 텍스트)
    msg = claude.messages.create(
        model="claude-opus-4-8",
        max_tokens=1024,
        messages=[{"role": "user", "content": event["text"]}],
    )
    reply = msg.content[0].text

    # 3) 대기 메시지를 실제 답으로 교체
    app.client.chat_update(
        channel=event["channel"], ts=placeholder["ts"], text=reply,
    )

if __name__ == "__main__":
    SocketModeHandler(app, os.environ["SLACK_APP_TOKEN"]).start()  # xapp
```

- 이게 **가장 작은 AI 봇**이다. `event["text"]`를 클로드에 넘기고 답을 올린다.
- **맥락 추가**: 스레드 히스토리(`conversations.replies`)나 내 문서를 `content`에 함께 넣으면 "내 자료로 답하는" 봇이 된다(§3).
- **긴 답**: 스트리밍으로 받아 `chat_update`를 반복하면 타이핑되듯 보인다 [13].
- Slack 공식 샘플(`bolt-python-ai-chatbot`)도 Anthropic/OpenAI를 갈아끼우는 구조를 제공한다 [14].

## 9. 한 단계 위 — 도구를 쓰는 에이전트

위 봇은 "묻고 답"만 한다. **에이전트**는 한 발 더 나아가 **스스로 도구를 호출**한다(DB 조회, 파일 읽기, 배포 트리거). 2026년엔 이를 위한 길이 정리됐다 [11][9].

- **Claude Agent SDK**: 클로드로 에이전트를 만드는 공식 프레임워크. **MCP 도구 연동 + 대화 관리**를 제공 [해외 Agent SDK].
- **Claude Managed Agents + Slack**: 에이전트가 **세션 이벤트 스트림을 읽어** 진행 상황을 Slack에 중계하고, 작업이 끝날 때까지 루프 돌다 **최종 요약**을 올린다(쿡북의 데이터 분석 봇) [11].
- **MCP 연동**: 봇이 GitHub·노션·DB를 **표준 방식으로** 다룬다(→ [[MCP(Model Context Protocol)]]). Slack 자체도 2026년 **MCP 클라이언트·스킬·에이전틱 Slackbot**을 추가했다 [9].

1인에게 의미: "요약·답변"을 넘어 **"조사해서 처리까지"** 하는 봇으로 진화. 단, 자율성이 커질수록 **승인 게이트·권한 제한**이 중요해진다(§11).

## 10. "나에게 유익한가?" — 정직한 판정 프레임

가장 궁금해한 부분. 봇은 만능이 아니다. **작업 성격**으로 판단하자. 아래 4축에서 **높음이 2개 이상이면 만들 가치**가 있다.

| 판정 축 | 높음(=봇 이득) | 낮음(=봇 손해) |
|---|---|---|
| **반복 빈도** | 매일/매주 반복 | 어쩌다 한 번 |
| **정보 과부하** | 읽을 게 많아 시간 잡아먹음 | 양이 적음 |
| **즉답 가치** | 빨리 알수록 이득(에러·문의) | 급하지 않음 |
| **오류 허용도** | 초안·1차 분류(사람이 검수) | 100% 정확 필수(법·회계·결제) |

**유익한 1인 시나리오**
- 고객 문의가 꾸준히 옴 → **1차 분류·초안 답변 봇**(반복·즉답↑).
- 혼자 운영하는 서비스 → **에러 요약·매출 질의 봇**(과부하·즉답↑).
- 리서치·콘텐츠 많이 함 → **요약·초안 봇**(과부하↑, 오류허용↑).

**유익하지 않은 경우(솔직히)**
- 트래픽·문의가 거의 없음 → 만들 시간이 더 아깝다.
- 출력이 **항상 100% 정확**해야 함(자동 송금·계약) → LLM 부적합, 규칙/사람으로.
- 맥락이 매번 완전히 달라 재사용 안 됨 → 그냥 클로드 앱에서 직접 물어보는 게 낫다.

> 한 줄 기준: **"이 일을 내가 매주 반복하고, 80% 품질의 초안이라도 시간이 크게 줄면" → 봇이 이득.** "가끔 하고, 틀리면 큰일" → 봇 말고 사람/규칙.

한 가지 더 현실적인 조언: **봇을 만드는 비용도 계산에 넣자.** 가장 작은 요약 봇은 한두 시간이면 띄우지만, "도구 쓰는 에이전트 + 승인 흐름 + 모니터링"까지 가면 며칠이 든다. 그래서 권하는 순서는 ①클로드 앱/웹에서 **그 작업을 손으로 며칠 해보고**(정말 반복되는지, 프롬프트가 안정적인지 확인) → ②반복이 확인되면 **요약/Q&A 같은 가벼운 봇**으로 자동화 → ③손이 많이 가는 핵심 작업만 **에이전트로 승격**이다. 즉 "봇부터 만들기"가 아니라 "**수동으로 검증한 반복만 봇으로**" 옮기는 게 1인의 시간을 아끼는 길이다. 만들고 안 쓰는 봇이 가장 비싸다.

## 11. 비용·리스크 — 공짜가 아니다

- **토큰 비용**: 클로드 API는 입력·출력 토큰으로 과금된다(2026-06 기준 Opus 4.8 입력 $5/출력 $25 per 1M, 모델별 상이). 긴 맥락(스레드 전체)을 매번 넣으면 비용↑ → **필요한 만큼만** 컨텍스트 주입, 캐싱 활용 [해외 가이드].
- **환각(hallucination)**: LLM은 그럴듯한 거짓을 만든다. **지식 Q&A는 출처 제시 + "모르면 모른다"** 를 강제하고, 중요한 답은 사람이 검수 [6].
- **권한·프롬프트 인젝션**: 외부 입력(문의·웹문서)에 "토큰 유출해라" 같은 악성 지시가 숨을 수 있다. 봇이 도구(배포·DB)를 쓸수록 위험↑ → **최소 권한 + 승인 게이트 + 신뢰 경계**(MCP 보고서의 보안 원칙과 동일) [9].
- **3초·지연**: 응답이 느릴 수 있으니 대기 메시지·스트리밍으로 UX 보완 [13].
- **운영 부담**: 봇도 소프트웨어다. 죽으면 재시작, 토큰 만료·요금 모니터링 필요.

## 12. 시작 추천 — 가장 ROI 높은 첫 봇

고민되면 **"요약 봇"**부터. 환각 위험이 낮고(요약은 원문 기반), 매일 쓰며, 만들기도 쉽다 [7][10].

```
1단계  앞 편의 Bolt 봇(Socket Mode)에 anthropic 설치
2단계  @봇 멘션 → 스레드 내용(conversations.replies) 수집
3단계  "이 대화 핵심 3줄 + 결정사항/할 일" 프롬프트로 클로드 호출
4단계  결과를 스레드에 게시 (대기 메시지 → chat_update)
5단계  쓸 만하면 분류·Q&A·에이전트로 확장
```

**체크리스트**
- [ ] 3초 함정 대비(즉시 ack/대기 메시지 + 비동기) [13].
- [ ] 컨텍스트는 **필요한 만큼만**(토큰 비용).
- [ ] 지식 Q&A는 **출처 제시 + 모르면 모른다**.
- [ ] 도구 쓰는 봇은 **최소 권한 + 승인 게이트** [9].
- [ ] API 키·토큰은 환경변수, 절대 평문 커밋 금지.
- [ ] "이 작업이 §10 4축에서 정말 이득인가" 자문 후 착수.

## 한눈 요약
- AI 봇 = Slack 안의 **실무자급 보조**(요약·Q&A·분류·운영·개인비서) [6][9].
- 구조: **Slack(입출력) + 클로드(두뇌) + 내 코드/도구(손발)** [11].
- 만들기: **Bolt + anthropic SDK**, 핵심 함정은 **3초 타임아웃**(즉시 ack+비동기) [12][13].
- 유익함은 **반복·과부하·즉답·오류허용** 4축으로 판정 — 2개↑ 높으면 이득.
- 비용·환각·권한을 설계에 반영. **첫 봇은 요약 봇**이 가장 안전한 본전 [7][10].
